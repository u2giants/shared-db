- Root cause of the silent outage: `plm.import_master_data()` set `status='failed'` then re-raised,
  so the aborted transaction rolled the failed row back; and the upstream 502 fails in `fetchJson()`

  `OnFailure=plm-sync-alert.service`; unit tests in `tools/sync-plm-master-data.test.mjs`.

## Environment facts worth keeping
- The Supabase **MCP is bound to prod and runs as `supabase_read_only_user`** — it can read prod but
  cannot write/execute functions or reset branches. Writes go through the migration workflow.
- Migrations ship via `gh workflow run shared-supabase-migrations.yml -r <branch|main> -f target=<preview|production> -f mode=<dry-run|apply>` (CI holds the creds). CI `validate` is path-filtered to `supabase/migrations/**`.
- To run SQL against the **preview** branch from a workstation: `op run` with the preview
  `POSTGRES_URL` (item id `ou6avbeyier7tifwqxup36ih4a`) but **use the session pooler (port 5432)** —
  the transaction pooler (6543) collides on the CLI's fixed prepared-statement name, and the direct
  host is IPv6-only/unreachable.
- Data migrations that call `merge_factory` (or anything that RAISEs on missing rows) MUST be
  guarded (`if exists(...) then ...`) so they no-op on the data-less preview branch and don't break
  future preview resets.
