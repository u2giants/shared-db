## 4. `plm.sample_shipment_item` structure (the restore template)

Columns: `shipment_item_id_pk int NOT NULL` (GENERATED BY DEFAULT AS IDENTITY),
`sample_id_fk int NOT NULL`, `box_id_fk int NULL`, `factory_group_id_fk int NULL`,
`leg_type varchar NULL`, `added_date timestamptz NULL`, `added_user varchar NULL`.

Constraints: PK on `shipment_item_id_pk`; FK `sample_id_fk → plm.sample` (ON UPDATE
CASCADE, ON DELETE CASCADE); FK `box_id_fk → plm.sample_box` (ON UPDATE CASCADE, ON
DELETE SET NULL). Indexes: primary key only. No unique constraint on
`(sample_id_fk, box_id_fk)` yet.

Grants on the `dflow` sample cluster: `postgres` only — the tracking service
connects through the pooler as the `postgres` role (Sequelize), not PostgREST, so
no `anon`/`authenticated`/`service_role` grants or RLS are required, matching the
six sibling tables restored on 2026-07-21.

## 5. Consumer-contract confirmation (answers §15 Q1)

`popcre/designflow-tracking` `models/sample.model.js` (`proveFactoryToNyc`) uses
`sample_shipment_item` as **current box membership** — it reads a single membership
for `(sample_id_fk, box_id_fk, leg_type)` and fails closed (HTTP 409) when the
model/table is unavailable. Its own comment states a future
`UNIQUE(sample_id_fk, box_id_fk)` constraint "will make membership unambiguous."
So the table is current-membership (not historical legs), and the uniqueness
constraint is the app's own intended design — no product decision blocks this
foundational tranche.
